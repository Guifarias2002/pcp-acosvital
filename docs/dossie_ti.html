Status geral: O sistema passou por revisão de hardening completo. Todas as rotas de API exigem autenticação JWT, 100% das queries usam parâmetros vinculados (imune a SQL injection), logs de acesso estruturados estão ativos e o direito ao esquecimento (LGPD Art. 18) foi implementado. Os 4 itens de atenção são operacionais (HTTPS, separação dev/prod, política de senha, rotação de chave JWT) e dependem de configuração de infraestrutura pela equipe de TI.
1 · Checklist de Requisitos TI — Status por Item
  • ✓
    Autenticação robusta JWT HS256 (8h), cookie HttpOnly + Secure em produção, fallback Authorization header. Rate limit: 10 tentativas/15 min por IP com cabeçalho Retry-After. Senhas via PBKDF2-SHA256 (timing-safe).
    Implementado
  • ✓
    Autorização por perfil (RBAC) Cada rota valida user.is_staff, user.setor e user.perfil. Operadores só acessam o setor próprio. Admins têm acesso irrestrito. Nenhuma rota expõe dados sem autenticação.
    Implementado
  • ✓
    Prevenção de SQL Injection 100% das queries usam template literals parametrizados do driver postgres. Zero concatenação de string em SQL. Validado em todas as 20 rotas de API.
    Implementado
  • ✓
    Validação de entrada nas fronteiras IDs convertidos e validados como inteiros positivos. Enums verificados contra listas fixas. Upload: máx. 10 MB, extensões JPG/PNG/PDF/GIF/WEBP. Strings truncadas em comprimentos máximos.
    Implementado
  • ✓
    Logs de acesso estruturados (LGPD/TI) Tabela auditoria_acesso: registra usuario_id, username, método HTTP, rota, ação, IP e timestamp para toda mutação sensível. Endpoint GET /api/auditoria com paginação e filtros para admins.
    Implementado
  • ✓
    Log de auditoria de login Tabela auditoria_login: registra tentativas de autenticação (sucesso e falha) com IP e timestamp. Permite detectar ataques de força bruta retroativamente.
    Implementado
  • ✓
    LGPD Art. 18 — Direito ao esquecimento Endpoint DELETE /api/usuarios/:id: anonimiza nome, e-mail e credenciais do usuário preservando o histórico de movimentações. Registra a anonimização no log de auditoria. Admins não podem excluir a si mesmos.
    Implementado
  • ✓
    Proteção contra XSS Next.js escapa automaticamente todo output no React. Nenhum dangerouslySetInnerHTML detectado. CSP pode ser adicionado via headers no next.config.js (recomendado pelo TI).
    Implementado
  • ✓
    Rate limiting em mutações Módulo rateLimit.ts: limite configurável por IP nas rotas de ação. Login: 10 req/15 min. Impede abuso automatizado e enumeração de recursos.
    Implementado
  • ✓
    Integridade de banco — constraints completas Rota POST /api/setup/hardening (idempotente): aplica NOT NULL, DEFAULT, UNIQUE, CHECK, FK (CASCADE/SET NULL), 20+ índices compostos, triggers de atualizado_em e recálculo de valor_total.
    Implementado
  • ✓
    Upload seguro de comprovantes Validação de tamanho (máx. 10 MB) e extensão antes do envio ao Supabase Storage. Path com timestamp. Sem acesso ao sistema de arquivos local do servidor.

  • ✓
    Backup com sobrescrita controlada Backup diário (backup_diario.xlsx) e backup de código (sistema_pcp.zip) sobrescrevem sempre o mesmo arquivo na pasta de rede. Sem proliferação de arquivos datados.
    Implementado
  • ✓
    Tratamento de erros sem vazamento de stack Blocos catch em todas as rotas retornam mensagens genéricas ao cliente. Stack traces são logados apenas no console do servidor, nunca expostos ao browser.
    Implementado
  • ✓
    Dependências auditadas npm audit executado. axios atualizado para versão sem CVEs. Único aviso restante: glob via eslint-config-next (dependência de dev, não afeta produção).
    Auditado
  • !
    HTTPS obrigatório em produção Cookie configurado com secure: true apenas quando NODE_ENV=production. O TI deve garantir que o servidor rode atrás de HTTPS (nginx/IIS com certificado TLS). Sem HTTPS, o cookie não é enviado em produção.
    Ação TI
  • !
    Separação dev / produção Recomenda-se bancos separados para dev e prod, com DATABASE_URL distintas. TI deve provisionar ambiente de homologação isolado antes do go-live.
    Ação TI
  • !
    Política de senhas para novos usuários Cadastro de usuários feito manualmente pelo admin. Recomenda-se política mínima: 8+ caracteres, sem senhas comuns. O hash PBKDF2-SHA256 é forte; a política de criação é responsabilidade do admin.
    Atenção
  • !
    Rotação do JWT_SECRET A chave JWT é estática no .env.local. Recomenda-se rotação a cada 90 dias ou em caso de suspeita de comprometimento. A rotação invalida todos os tokens ativos (usuários precisam re-logar).
    Atenção

5 · Banco de Dados — Tabelas de Auditoria
Estrutura das tabelas de log
-- Log de tentativas de login
CREATE TABLE auditoria_login (
  id        BIGSERIAL PRIMARY KEY,
  username  TEXT NOT NULL,
  ip        TEXT,
  sucesso   BOOLEAN NOT NULL,
  criado_em TIMESTAMPTZ DEFAULT NOW()
);

-- Log estruturado de ações
CREATE TABLE auditoria_acesso (
  id         BIGSERIAL PRIMARY KEY,
  usuario_id INTEGER,
  username   TEXT NOT NULL,
  metodo     TEXT NOT NULL,
  rota       TEXT NOT NULL,
  acao       TEXT,
  ip         TEXT,
  criado_em  TIMESTAMPTZ DEFAULT NOW()
);
Execute POST /api/setup/hardening como admin para criar estas tabelas no banco.
Consulta dos logs (via API)
-- Acessar via GET /api/auditoria
-- Parâmetros disponíveis:
--   ?usuario_id=5
--   ?rota=/api/item
--   ?de=2026-06-01
--   ?ate=2026-06-30
--   ?pagina=1&limite=50

-- Ou direto no Supabase:
SELECT username, acao, rota, ip,
       criado_em
FROM auditoria_acesso
WHERE criado_em >= NOW() - INTERVAL '7 days'
ORDER BY criado_em DESC;

7 · Cronograma Sugerido
  • Semana 1
    Reunião de alinhamento técnico
    Apresentação da arquitetura, walkthrough dos endpoints, revisão das variáveis de ambiente, definição do ambiente de homologação. Entrega deste dossiiê ao TI.
  • Semana 1–2
    Setup do ambiente de homologação
    TI provisiona servidor (Node.js 18), configura HTTPS, cria banco de homologação separado, define .env de produção. Desenvolvedor dá suporte técnico.
  • Semana 2–3
    Hardening e testes de segurança
    Admin executa POST /api/setup/hardening no banco de prod. TI realiza code review, testes de penetração básicos, validação dos logs de auditoria.
  • Semana 3–4
    Testes de usuário (UAT)
    Usuários-chave de cada setor validam o fluxo operacional. Admin valida backups. TI monitora logs durante o período.
  • Semana 4
    Go-live e congelamento de escopo
    Sistema vai para produção. Custódia passa ao TI. Qualquer nova funcionalidade segue processo formal de solicitação e aprovação pelo TI.

    Instalar Node.js 18 LTS no servidorVerificar: node -v deve retornar v18.x ou superior.

  • 3
    Criar .env.local de produçãoDATABASE_URL (banco prod), JWT_SECRET (mín. 32 chars aleatórios), SUPABASE_URL, SUPABASE_SERVICE_KEY, NODE_ENV=production.
  • 4

    Configurar HTTPSnginx ou IIS na frente do Next.js (porta 3000). Certificado da CA interna ou Let's Encrypt.
  • 7
    Configurar pasta de rede para backupsZ:\Ordens de Serviço - IAPP\NOSSO SISTEMA acessível pelo processo Node.js. Testar com o botão de backup.
